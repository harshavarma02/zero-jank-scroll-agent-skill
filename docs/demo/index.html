Before and after: fixing a laggy sticky-scroll section

The two demos use the same content and visual concept. The before version performs avoidable work during native scrolling. The fixed version lets CSS own pinning and changes the right-side panel only when the next left-side control reaches the sticky top line.

Before400vh track, global listener, repeated layout reads and DOM writesAftersemantic steps, CSS sticky, top-aligned IntersectionObserver

How to compare fairly

  1. Use the same browser, viewport, CPU throttling, and input device.
  2. Start at the hero and perform the same slow scroll, fast flick, click-to-step, and reverse-scroll sequence.
  3. Keep the on-page counters visible.
  4. Do not add artificial video stutter.
Evidence boundary: the counters demonstrate architecture-level work performed by these examples. They do not claim a universal FPS improvement for every device or project.
