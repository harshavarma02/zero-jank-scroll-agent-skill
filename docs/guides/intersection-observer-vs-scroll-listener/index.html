Architecture decision

IntersectionObserver vs scroll listener for scroll-driven UI

The correct choice depends on whether the interface needs a discrete identity or a continuous value. Treating every scroll interaction as continuous progress is a common source of unnecessary complexity.

Use IntersectionObserver for discrete activation

Choose an observer when you need to know which chapter, capability, product section, or navigation item is active. Observe real elements, align the observer activation line with the visual sticky offset, and update state only when the identity changes.

Use continuous progress only when the visual requires it

A progress bar, path drawing, parallax amount, or scrubbed scene may need a value on every animation frame. Prefer a CSS scroll timeline when browser requirements permit, an existing animation primitive such as a MotionValue, or one requestAnimationFrame-coalesced controller. Do not feed the value through broad component state.

Why passive is not enough

A passive scroll listener can improve input handling when the listener never prevents default behavior, but it does not make expensive JavaScript, layout reads, DOM writes, or component rerenders inexpensive. The work still has to execute.

Top-line versus center activation

Use the viewport center only when the visual design says a chapter becomes active at the center. For sticky step interfaces, users often expect the panel to change when the next left-side heading reaches the same top line as the sticky panel. Share one CSS offset between the sticky element, scroll target, and observer root margin.

Identity problem: observer. Progress problem: scroll timeline or controlled frame value. Timeline choreography: a dedicated animation timeline such as ScrollTrigger.
