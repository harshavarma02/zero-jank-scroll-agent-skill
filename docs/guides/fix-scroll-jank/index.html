Diagnostic guide

How to fix scroll jank without guessing

Scroll jank is a symptom, not a single bug. It can come from main-thread work, forced layout, large paints, component rerenders, media decoding, sticky containment, or animation smoothing that intentionally trails the input. The fastest path is to reproduce the exact failure and inspect causes in a stable order.

1. Define what “laggy” means

Record whether the failure is dropped visual frames, delayed state changes, skipped chapters, sticky detachment, layout jumps, late input response, or a scrubbed animation that trails the pointer. Different symptoms need different fixes.

2. Inspect the hot scroll path

Search for global scroll listeners, repeated DOM queries, logging, parsing, framework state setters, synchronous storage, and layout reads such as bounding rectangles or offsets. A discrete four-step interface should not calculate continuous progress on every event.

3. Check read/write ordering

A repeated write-read-write sequence can force the browser to resolve style and layout more often than necessary. Batch stable measurements, prefer observers for identity changes, and keep continuous values outside broad reactive state.

4. Inspect paint and hidden work

Large backdrop filters, masks, blur, oversized shadows, fixed backgrounds, autoplay video, canvas loops, WebGL scenes, and inactive Lottie animations may continue consuming resources while hidden. Pause inactive work and profile the actual paint regions.

5. Verify the fix

Repeat the same wheel, trackpad, keyboard, reverse-scroll, resize, zoom, and reduced-motion sequence. Compare traces and render counts. Report what improved and what remains unmeasured.

For discrete step activation, start with semantic steps, CSS sticky positioning, one IntersectionObserver, and state changes only when the active identity changes.
